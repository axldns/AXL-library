Classpublic class Ldr
InheritanceLdr Object

It easily allows you to load/unload/save files, instantiate AS3 Objects out of file contents, keep your server and app files in sync.
  • loads files from local and remote directories - supports relative paths on both (general rules apply)
  • looks up for user defined alternative directories (if file not found) in user defined order
  • stores to hdd (AIR) on various conditions, including date comparison, file-directory-extension filters, user defined filters.
  • Supports multi queues with various and mixed types of addressing (arrays, vectors with/or File class instances, strings of paths and sub-paths
  • provides detailed info about progress and full controll on loading process (pause, stop, resume, queues/files re-index)
  • processes commonly known data to expose usable forms (images, sounds, xmls, json) but leaves the rest un-touched.
All that within one line of code, more ! with just one function! That's it. Don't need to learn anything else. Easily accesible without instantiation, anywhere in your code just like Ldr.load() All objects are also super simple to access with just one command, anywhere from the code just like Ldr.getMe() This Ldr assumes you know what you're doing, and if you're dealing with files, you want to call 'file.txt' rather than 'file'. You can load and get file.png, file.jpg, file.atf, file.xml, file.txt and call them all whenever you like without worrying of anything else. Properly set up, even three lines of code can make a robust solution for assets update managed fully server side (mobile apps). Full controll on loading process (pause, resume, queues re-index) error handling, verbose mode XML may contain two types of nodes: files and file. file nodes should hold only filename or sub-path. files nodes should be a list of file nodes. files also can have an attribute dir Such an attribute will be a prefix to all its sub-nodes. 
	 <files dir="/assets">
	 	<file>/sounds/one.mp3</file>
	 	<files dir="images">
	 		<file>two.png</file>
	 		<file>two.jpg</file>
	 	</files>
	 </files>
	 <file>/config/workflow/init.xml</file>
This would queue: /assets/sounds/one.mp3 /asset/images/two.png /assets/images/two.jpg /config/workflow/init.xml

public static function addToCurrentQueue(resourceOrList:Object):int

adds path, file or list to load to current queue. This is not preffered method for adding elelments to queue since Ldr.load accepts arrays, vectors, xmls. Use addToCurrentQueue when you need to inject to current. load()method 
public static function load(resources:Object = null, onComplete:Function = null, individualComplete:Function = null, onProgress:Function = null, pathPrefixes:Object, loadBehavior:Object, storingBehavior:Object, storeDirectory:Object, timeOutMS:int = 0):int

Loads all elements/files one by one from array of paths, subpaths or single url. Checks for alternative directories, stores loaded files to directories (AIR only). It controlls loading same asset twice. Use Ldr.unload to remove previously loaded and instantiated files.

Parameters

resources:Object (default = null) — : Basic types: String, File, XML, XMLList or collections: Array, Vector of basic types. Basic elements must always point to file with an extension exposed. eg.: ["/assets/images/a.jpg", "http://abc.de/fg.hi"] Exception is the File class instance; If it points to directory, whole directory will be scanned recursively and every file found will be added to queue. Resources can be mixed together and nest to the reasonable level of depth - lists are parsed recursively too! Check class description to see examples including XML patterns.
 
onComplete:Function (default = null) — : function to execute once queue is done. this suppose to execute always, regardles of issues with particular asssets. Exception is calling Ldr.load(null, ..anyArgs) while queue is already loading. This prevents before overwriting listeners and/or dispatching when actually not done.
 
individualComplete:Function (default = null) — : function(loadedAssetName:String) this function may not be executed if loader can't resolve either prefix or path to resource, but it is executed if loading failes - Ldr.getAny(loadedAssetName) would return null in this case. As long as you pass correct resource types, function should always get executed. individualComplete function will get executed only once for one item, regardless of how many prefixes has been checked for it before.
 
onProgress:Function (default = null) — : function which should accept two arguments:
  • Number - bytesLoaded / bytesTotal of current asset
  • String - name of currently loaded element
To get detailed info of queue(s) status, whenever you need it query Ldr.num^ getters Don't worry - there are no calculations on query time - values are being updated only when queue(s) status changes.
 
pathPrefixes:Object (default = NaN) — These values will be prefixes for resources defined elemements. Consieder it as a member of simplified equasion where singleElementUrl = pathPrefixes[i] + resources[j] Double joints like "//" or "\\" or even missing separators should get fixed automatically. If loading fails, pathPrefixes index will keep increassing until a) element will get loaded or b) pathPrefixes index will reach maximum value. In both cases pathPrefix index will roll back to 0 for the next element. Process applies to every queued item, therefore use your pathPrefixes wisely.
  • resource - Object is parsed simmilar way as resources argument is but: 1 pathPrefixes must not point to files (directories only) 2 directories are not scanned recursively (simple hook)
  • null will match pathList only (empty prefix adds itself)
  • Ldr.defaultValue uses
 
loadBehavior:Object (default = NaN) —
  • Ldr.behaviors.loadOverwrite - any previously loaded elements of the same name as particular element (requested to load) will be unloaded and removed from memory. All loaded elements will be instantiated.
  • Ldr.behaviors.loadSkip - any previously loaded elements of the same name will cause to skip loading queue element. onIndividualComplete will get executed, counters updated, queue will continue as usual.
  • Ldr.downloadOnly (AIR) - none of loaded elements will be instantiated. Saving files on disc will proceed according to storingBehavior
  • function(colidingFilename:String):String - custom filter allows to define what to do name of element requested to load colides with already loaded resources. return null to skip loading the file. return String to assign either new or the same name. Any name colision after this point will cause to unload existing resource and new will take its place.
  • Ldr.defaultValue will use
 
storingBehavior:Object (default = NaN) — (AIR):
  • Ldr.Behaviors.default will perform using values.
  • RegExp particular resource will be stored in storeDirectory if its URL matches your your RegExp. Good scenario to store network updated files only by passing /^(http:|https:|ftp:|ftps:)/i or to filter storing by extensions. Pass /./ to store/overwrite all files from this queue. Define storeDirectory as null to disable storing files from this queue.
  • Date or Number where number is unix timestamp. This stores files if a) file does not exist in storeDirectory yet b) your date is greater than existing file modification date.
  • function(existing:File, loadedFrom:String):File - This function would be called for every element loaded from address different to storeDirectory. This allows to decide if (and where) file should get saved/overwritten. By dispatch time you're receiving an empty pointer to directory resolved according to storeDirectory + resource[i] (sub path) File existance, contents and address are not being verified and may be result of any activty but you may want to use them as your criteria. Closure excepts a pointer to file in any storable directory as an output. null or incorrect values will withold this file from storing. Performance is on you in this case.
 
storeDirectory:Object (default = NaN) — (AIR): Defines where to store loaded files if storingBehavior allows to do so. If value can't be interpreted as storable directory - no storring will happen, regardless of storingBehavior assigned. E.g.: Ldr.load("/assets/image.png",null,null,null,"http://domain.com",File.applicationStorageDirectory,/./); would load: http://domain.com/assets/image.png would save: app-storage:/assets/image.png
  • Ldr.defaultValue uses
  • String tries to resolve path
  • File tries to resolve path
  • null and/or other incorrect values - disables storing
 
timeOutMS:int (default = 0) — Defines time (ms) after which particular element request will move to the next path prefix (if specifed) or to the next element in queue IF server does not response at all. If your server is down, response may not come immediately but after browser defined timeout or (AIR) urlRequest.idleTimeout value. Use this parameter to shorten your response awaiting time. This limits OPEN time, not loading time.

Returns
int —
  • -2 if there are no resources specified and no queues to start
  • -1 if there are no resources specified and queue is already started
  • ID of the queue
